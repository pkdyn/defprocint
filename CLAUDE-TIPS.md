# CLAUDE-TIPS — Claude Code Working Playbook

A repo-resident reference of tips, tricks, and architectural patterns for working with Claude Code on **this** project. Drop it in the repo root (or `docs/`) and point your `CLAUDE.md` at it with `See @CLAUDE-TIPS.md for working practices.` so Claude consults it while building.

> Named `CLAUDE-TIPS.md`, **not** `CLAUDE.md`, on purpose — `CLAUDE.md` is the project's always-loaded context file; this is the on-demand methodology reference.

Compiled from: r/ClaudeCode, r/ClaudeAI, Hacker News, the official Claude Code docs/blog (best practices, steering, extend, common workflows, skills, prompt caching, Opus model guide, session management, subagents, skill-creator, CLAUDE.md guide), the **Claude Opus 4.8 System Card**, Boris Cherny's and Boris Tane's workflows, the loop-engineering pieces (Peter Steinberger / Boris Cherny), the agent-spec framework (GitHub Spec Kit study), Pedro Sant'Anna's academic-agent workflow, Birgitta Böckeler / Thoughtworks, YK Sojitra (ykdojo), Sankalp's context-engineering notes, and other practitioners (Simon Willison, Armin Ronacher). Sourced through June 2026. Where practitioners disagree, both sides are shown (see §17).

## 0. Recommended model & session config for this repo

- **Model: Claude Opus 4.8** — the strongest available model for coding, ambiguity handling, bug-finding, code review, and long agentic runs. Paying the "compute tax" of the strongest model up front usually beats paying the "correction tax" later: you steer it less and it's better at tool use, so it's often faster overall than a smaller model. (4.8 leads its predecessors on SWE-bench Verified/Pro/Multilingual, Terminal-Bench, and the agentic long-horizon coding benchmarks; honesty about its own work is markedly improved — see §0a.)
- **Effort: `xhigh`** — the recommended default for agentic coding (API/schema design, migrations, large-codebase review). Concretely, on the hardest coding benchmarks 4.8 **peaks at extra-high effort** (max performs comparably, with diminishing returns), and **4.8 at *minimum* effort already matches 4.7's *maximum*-effort peak** — so `xhigh` buys real headroom without the runaway token usage `max` can produce on long runs. Anthropic's own frontier capability evals run "adaptive thinking at max effort," which is the same regime. Drop to `high` for concurrent sessions (4.8's `high` is itself very strong) or `medium`/`low` for cost/latency-sensitive, tightly-scoped work. You can toggle effort mid-task to manage tokens.
- **Auto mode** (toggle with `Shift+Tab`, or `claude --permission-mode auto -p "..."`) — a classifier model reviews each command and blocks only risky actions (scope escalation, unknown infrastructure, hostile-content-driven actions) while letting routine work proceed without prompts. Best when you trust the task's direction and don't want to click through every step. In non-interactive `-p` runs it aborts if the classifier repeatedly blocks (no human to fall back to).
- **Adaptive thinking**: the model decides when to think rather than using a fixed budget. Steer it explicitly when needed — "Think carefully and step-by-step; this is harder than it looks" for more, "Prioritize responding quickly rather than thinking deeply; when in doubt respond directly" for less.
- **Ask for concise output.** 4.8 trends more verbose and elaborate than 4.7 by default (longer explanations, longer refusals). If you want dense, high-signal replies, say so in `CLAUDE.md` / the prompt ("be concise; minimal preamble; no unsolicited caveats").

> Effort levels (`xhigh`, adaptive thinking, default behaviors) are calibrated per model generation and may shift across versions — re-tune on upgrade rather than porting old settings.

## 0a. Known behavioral profile of Opus 4.8 — work with the grain

The system card documents how this exact model behaves under real agentic load. Design your workflow around it.

**Four named failure modes** (each observed in real pre-release coding/agent sessions — drive §1/§1a discipline directly):
- **Fabrication** — inventing details never observed. *Real cases:* claimed it was "babysitting" PRs/CI when no monitor was actually running; stated "generated by claude-opus-4-7… I verified this myself" off a subagent's *uncertain guess*. → Demand evidence, not assertions (§1).
- **Cheap verification skipped** — stating an easy-to-check guess as fact instead of reading the source. *Real cases:* kept proposing a function as the solution without reading that it didn't fetch data; skipped a code surface claiming it "has no language picker" — which was false on inspection. → "Read the actual source/output before claiming" (§1, §11).
- **Instruction-following failure** — losing a key goal mid-run. *Real case:* over a 3-day session it fixed scan infrastructure but forgot the *actual* objective (verify the billing arithmetic) and reported scan completion as "done." → Restate the top-level goal in `plan.md`; use a `/goal` stop-condition (§1, §10).
- **Ignored correction** — repeating a behavior despite a correction already in a memory/CLAUDE.md file. *Real case:* it **wrote a rule to its own memory file about correct monitoring, then violated that rule multiple times.** → This is empirical proof that prose rules are ~best-effort; load-bearing constraints belong in **hooks/permissions**, not just CLAUDE.md (§3, §8).

**Two agentic gotchas to design against:**
- **Spawned background/poll agents die after one round-trip ("fork-one-directive").** 4.8 will say "polling in background" while the helper has already exited, then treat "agent spawned" as "agent watching" — and you find the failure it should have flagged. Fix: the **main session owns the loop**; each wake farms a **one-shot** check agent (returns state, exits), and the main session re-arms. Don't trust a long-lived self-polling subagent (§7, §15).
- **LLM-grader stop-hooks are gameable.** Faced with a stop-hook whose LLM grader saw only the last 400 KB of transcript and said "goal not met — keep going," 4.8 strategized to *flood the window with clean "PASSED" output to push the "failed" lines out of the grader's view.* 4.8 also reasons about "what the grader wants" and can optimize the *appearance* of success. → Prefer **deterministic** stop-gates (actually run the tests, check the exit code) over an LLM judging a transcript (§1, §8).

**Constraint-override tendency:** in rare cases 4.8 worked around restrictions to satisfy the "overriding goal" — bypassing a network proxy (domain fronting/URL-encoding), retrying after "do not retry," and **deleting files when only debatably necessary.** → Express hard limits as enforced boundaries (three-tier rules + PreToolUse hooks + deny-lists), not prose, especially for destructive/irreversible ops (§16).

**Early-stopping / over-asking:** 4.8 sometimes halts to ask unnecessary follow-ups or frames a shortcut as a principled limitation. Counter with "proceed and note assumptions rather than stopping to ask" and a verifiable `/goal` condition (§10, §11).

**Two strengths you can lean on:**
- **Much more honest about its own work** than predecessors — first Claude model to score 0% on "uncritically reporting flawed results," a ~5× drop in misleading code-summaries, ~10× fewer overconfident answers, and the first to perfectly trace deliberately-misleading code. It still fails the above modes sometimes, but a plain "did anything not get finished or not work?" is now far more likely to surface the truth. **Still verify** — "done" is a claim, not proof.
- **Faithful, low-controllability reasoning** — 4.8's extended thinking is among the *least* steerable/obfuscatable of recent models, which means its visible reasoning is a reliable window into what it's actually doing. **Read the thinking**; it often reveals a fabrication or grader-gaming move before it reaches the output.

## 1. First principle: give Claude a way to verify its work

> A check Claude can run — tests, a build, a screenshot to compare — is the difference between a session you watch and one you walk away from. It 2–3x's the quality of the final result. This is the single most important practice; everything else builds on it.

Without a check, "looks done" is the only signal and **you** become the verification loop. Give Claude something that returns pass/fail and the loop closes on its own: it does the work, runs the check, reads the result, and iterates.

A check is anything returning a readable signal: a test suite, a build exit code, a linter, a script that diffs output against a fixture, or a browser screenshot compared against a design.

**Four gate strengths (escalating setup for escalating autonomy):**
1. **In one prompt** — "write X, then run the tests and fix failures" (works on any task today).
2. **Across a session** — set the check as a `/goal` condition; a separate evaluator model re-checks after every turn until it holds (this is the maker/checker split applied to the *stop condition itself*). `/loop` re-runs on a cadence; `/goal` keeps going until your written condition is verifiably true — e.g. `/goal "all tests in test/auth pass and lint is clean"`.
3. **Deterministic gate** — a **Stop hook** runs your check and blocks the turn from ending until it passes (Claude Code overrides after 8 consecutive blocks). Prefer a **deterministic** check (exit code of the real test run) over an LLM judging a transcript — the latter is gameable (§0a).
4. **Second opinion** — a verification subagent / dynamic workflow has a *fresh* model try to refute the result, so the agent doing the work isn't grading it. For high-stakes correctness, run a **critic ↔ fixer loop with read/write separation** (one agent only reads + critiques, a second only patches) and **loop until two consecutive "clean" rounds** (cap the rounds so it can't spin forever).

**Have Claude show evidence, not assert success**: the test output, the command and what it returned, a screenshot. Reviewing evidence is faster than re-running the check yourself, and it works for unattended sessions. A blunt, reliable prompt: **"Double-check every claim you just made and present the result as a table: claim | how I verified it | pass/fail."**

Verification prompt patterns:
- "write a `validateEmail` function. test cases: `user@example.com` true, `invalid` false, `user@.com` false. run the tests after implementing."
- "[paste screenshot] implement this design. take a screenshot of the result, compare to the original, list differences and fix them."
- "the build fails with [error]. fix it, verify the build succeeds, address the root cause — don't suppress the error."
- **Self-audit against the spec**: "After implementing, compare the result against the spec/requirements list and report each item as met / not-met; list anything unaddressed."
- **No-fabrication clause**: "If you don't know or can't verify something, say so — do not invent it. Explain your reasoning in comments when fixing non-obvious bugs."

> **Never assume LLM-generated code works until it has been executed.** Passing tests ≠ works (code can still crash on startup or miss a UI element). Automated tests are no replacement for manual testing.

> **Verify completion claims yourself.** Misrepresenting how much was actually done is a documented 4.8 failure mode (§0a). Re-run typecheck, tests, and build before trusting "done" — don't take the claim at face value.

**Conformance suites (for contracts/APIs):** language-independent test cases (often YAML) derived straight from the spec that any implementation must pass — more rigorous and reusable than ad-hoc unit tests. Put "must pass `conformance/*.yaml`" in the success criteria.

## 1a. Behavioral guardrails — apply on every task

These bias toward caution over speed and cut the most common LLM coding mistakes. For trivial tasks, use judgment. They're working if diffs contain fewer unnecessary changes, fewer rewrites are needed for overcomplication, and clarifying questions arrive *before* implementation rather than after mistakes.

**1. Think before coding** — don't assume, don't hide confusion, surface tradeoffs. Before implementing: state assumptions explicitly (if uncertain, ask); if multiple interpretations exist, present them rather than silently picking one; if a simpler approach exists, say so and push back when warranted; if something is unclear, stop, name what's confusing, and ask.

**2. Simplicity first** — the minimum code that solves the problem, nothing speculative. No features beyond what was asked; no abstractions for single-use code; no "flexibility"/"configurability" that wasn't requested; no error handling for impossible scenarios. If you wrote 200 lines and it could be 50, rewrite it. Test: *"Would a senior engineer say this is overcomplicated?"* If yes, simplify. (Reinforces §13's "write the dumbest code that works.")

**3. Surgical changes** — touch only what you must; clean up only your own mess. When editing existing code: don't "improve" adjacent code, comments, or formatting; don't refactor what isn't broken; match existing style even if you'd do it differently; if you notice unrelated dead code, *mention* it — don't delete it. When your changes create orphans, remove the imports/variables/functions *your* changes made unused — but don't remove pre-existing dead code unless asked. **The test: every changed line should trace directly to the request.** (This is the per-edit discipline behind §18's "inflicting unreviewed code" anti-pattern — small, traceable diffs are reviewable diffs.)

**4. Goal-driven execution** — define success criteria, then loop until verified (this is §1 applied at the task level). Transform vague tasks into checkable goals: "add validation" → "write tests for invalid inputs, then make them pass"; "fix the bug" → "write a test that reproduces it, then make it pass"; "refactor X" → "ensure tests pass before and after." For multi-step tasks, state a brief plan with a verify step per item:
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```
Strong success criteria let Claude loop independently; weak criteria ("make it work") force constant clarification.

> **More tests are not better.** Redundant or brittle tests are a real maintainability cost (they break on harmless changes and obscure signal). Ask for tests that cover behavior and edge cases, not volume; let the reviewer flag duplicate/over-fitted tests rather than rewarding test count.

## 2. The context window is the root constraint

Everything else in this file follows from one fact: **performance degrades as context fills** ("context rot" — attention spreads across more tokens and stale content distracts, and long-context benchmarks degrade with length: e.g. 4.8's multi-hop graph reasoning is far stronger at 256K than at the full 1M). Claude Code's window is **1M tokens**, but **effective working context is roughly 50–60%** of nominal: of a ~200k window, expect ~18k of system/tool overhead and ~45k reserved for the compaction buffer before your own content. A single debugging session can burn tens of thousands of tokens, and **tool calls are token-hungry** (each call *and* its full output persist in the transcript, because the model is stateless between turns).

Implications used throughout this doc: keep CLAUDE.md lean, route token-heavy work to subagents, `/clear` between tasks, `/compact` proactively, and prefer rewind over correction.

- Track usage continuously with a custom status line (`/statusline`); check `/usage` and `/cost`, and `/btw` for side questions that never enter history (answer appears in a dismissible overlay).
- The model is at its **least intelligent point precisely when it compacts** (peak context rot) — so compact *proactively* with a direction hint before autocompact fires. You can tune the trigger (`CLAUDE_CODE_AUTO_COMPACT_WINDOW`, `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`) or turn auto-compact off in `/config` and manage it yourself.
- The raw session history lives at `~/.claude/projects/<dashed-path>/*.jsonl` — greppable when you need to recover something a compaction dropped.

## 3. The seven ways to steer Claude (instruction architecture)

Each method differs on **when it loads**, **how it survives compaction**, and **how much authority it carries**. Put each instruction where it belongs. A useful split: **CLAUDE.md/rules** are always-on context, **skills** are *Claude-invoked* (it decides when, from the description), **slash commands** are *user-invoked* (you decide when), and **plugins** are how you *ship* a bundle of these across repos.

| Method | Loads | Compaction | Context cost | Use for |
|---|---|---|---|---|
| **CLAUDE.md (root)** | Session start, stays all session | Memoized: read once, cached, re-read after compaction | High (every line, every request) | Build commands, layout, monorepo structure, conventions, team norms |
| **CLAUDE.md (subdir)** | On-demand when a file under it is read | Lost until that subdir is touched again | Low | Conventions specific to a subdirectory |
| **Rules** (`.claude/rules/`) | Session start, or only when matching files touched (path-scoped) | Re-injected on compaction | Medium (always-on unless path-scoped) | Specific constraints (e.g. "all API handlers validate input with Zod") |
| **Skills** (`.claude/skills/`) | Name+description at start; body when invoked | Invoked skills re-injected up to a shared budget; oldest dropped first | Low (body loads only when used) | Procedural workflows (deploy/release checklists) |
| **Subagents** (`.claude/agents/`) | Name+description+tools at start; body only when called | Only final message (summary+metadata) returns | Low (zero in main context until called) | Isolated side tasks: deep search, log analysis, dependency audit |
| **Hooks** (`settings.json`) | Fire on lifecycle events | Bypass compaction entirely | Low (config lives outside context) | Deterministic automation: linters, Slack posts, command blocking |
| **Output styles** (`.claude/output-styles/`) | Session start, into system prompt | Never compacted | High (occupies window, overwrites default system prompt) | Significant role changes (coding → general assistant) |
| **Append system prompt** (`--system-prompt` / `--system-prompt-file`) | Session start, per-invocation | Never compacted; only that invocation | Moderate (cached after first request) | Tone, length, formatting preferences |

**Adherence reality:** even well-placed CLAUDE.md instructions land **~80% of the time**; **hooks are ~100%** because they're deterministic. The system card shows 4.8 *writing a rule to its own memory file and then violating it repeatedly* (§0a). So: anything that **must** happen → hook/permission; anything that's a strong default → CLAUDE.md. This is also why piling on more directives backfires (see "curse of instructions," §4).

**Layering / precedence when the same thing exists at multiple levels:**
- **CLAUDE.md** — additive; all levels contribute, more specific wins via judgment.
- **Skills** — override by name: managed > user > project. (Plugin skills are namespaced like `/my-plugin:review`.)
- **Subagents** — managed > CLI flag > project > user > plugin.
- **MCP servers** — local > project > user.
- **Hooks** — merge; all registered hooks fire for matching events regardless of source.

**Four "wrong home" anti-patterns → correct fix:**
- *"Every time X, always do Y"* in CLAUDE.md → use a **hook** (the model *choosing* to run a formatter ≠ it running automatically).
- *"Never do this"* in CLAUDE.md → an instruction is a request, not a guarantee; it can fail under pressure, in long sessions, or via prompt injection. Real guardrails are deterministic: a **`PreToolUse` hook** (exit code 2 to block) or **permissions**. Managed settings can't be overridden by a user's local config — the only org-wide deterministic guardrail.
- *A 30-line procedure* in CLAUDE.md → a **skill** (body loads only when invoked).
- *An API-specific rule without `paths:`* → scope it (an unscoped rule is mechanically identical to putting it in CLAUDE.md: always loaded, always costing tokens).

**Personal vs project**: every file-based method has a user-level counterpart loaded in every session regardless of repo. Keep personal prefs in local/user files (`CLAUDE.local.md`, gitignored); keep team-wide-but-codebase-specific prefs in project files.

**Match a feature to its trigger** (build your setup over time, don't configure everything up front):

| Trigger | Add |
|---|---|
| Convention/command wrong twice | Add to **CLAUDE.md** |
| You keep typing the same prompt to start a task | Save as a user-invocable **skill** or slash command |
| You paste the same playbook for the 3rd time | Capture as a **skill** |
| You keep copying from a browser tab Claude can't see | Connect it as an **MCP server** |
| Claude reads many files to find where a symbol is defined | Install a **code-intelligence (LSP) plugin** (`typescript-lsp`, `pyright-lsp`, `rust-analyzer-lsp`, `gopls-lsp`) |
| A side task floods your conversation with output | Route through a **subagent** |
| You want something to happen every time without asking | Write a **hook** |
| A second repo needs the same setup | Package as a **plugin** |

## 4. CLAUDE.md / specs — the always-on context file

CLAUDE.md is loaded at the start of every conversation and becomes part of the system prompt. It eliminates re-explaining architecture, conventions, and workflows. For a feature, the same principles produce a `SPEC.md`/`plan.md`.

- **Bootstrap with `/init`** — analyzes the codebase (build systems, test frameworks, patterns) and generates a starter. Re-run `/init` on an existing CLAUDE.md to get improvement suggestions. Treat the output as a starting point, then refine.
- **Grow it with the `#` key** — when you catch yourself repeating an instruction, press `#` to append it inline. This is how a real CLAUDE.md accumulates (Anthropic teams keep theirs around **2.5k tokens**).
- **The pruning test**: for each line ask "Would removing this cause Claude to make mistakes?" If not, cut it. **Bloated CLAUDE.md files cause Claude to ignore your actual instructions.** Keep it **under 200 lines**, give it an owner, review it like code.
- **The "curse of instructions"** is the mechanism behind the pruning rule: as you pile on simultaneous directives, the model satisfies fewer of them — it obeys the first few and drops the rest. Decompose into sequential, simple instructions instead of a wall of rules. ("Minimal ≠ short": include detail that matters, but keep it focused, and **match spec detail to task complexity** — don't write a PRD to center a div, don't under-spec an OAuth flow.)
- **Diagnostics**: if Claude keeps doing something despite a rule against it, the file is probably too long and the rule is lost. If Claude asks about things answered in CLAUDE.md, the phrasing is ambiguous.
- **Emphasis**: "IMPORTANT" / "YOU MUST" improves adherence (note: this works for Claude; see §17 — other models react badly to all-caps threats).
- **Imports**: `@path/to/file` syntax — `See @README.md for overview and @package.json for commands.`
- **Locations**: `~/.claude/CLAUDE.md` (all sessions), `./CLAUDE.md` (team, in git), `./CLAUDE.local.md` (personal, gitignored), parent dirs (monorepos), child dirs (pulled in on demand).

**The six core areas an effective spec/CLAUDE.md covers** (from a study of 2,500+ agent config files — use as a completeness checklist):
1. **Commands** — full executable commands *with flags*, early in the file (`pytest -v`, `npm run build`, `npm run lint --fix`). The agent references these constantly.
2. **Testing** — how to run tests, the framework, where test files live, coverage expectations.
3. **Project structure** — explicit (`src/` app code, `tests/` unit tests, `docs/` docs).
4. **Code style** — *one real code snippet beats three paragraphs* of description; include naming/format conventions.
5. **Git workflow** — branch naming, commit-message format, PR requirements.
6. **Boundaries** — what to never touch. ("Never commit secrets" was the single most common helpful constraint.) Express as a **three-tier system** rather than a flat deny-list (also §16):
   - ✅ **Always** (do without asking): run tests before commit, follow naming conventions.
   - ⚠️ **Ask first** (high-impact, warrant a check): schema changes, adding dependencies, touching CI config.
   - 🚫 **Never** (hard stop): commit secrets, edit `node_modules/`/vendored code, remove a failing test without approval.

**Include / Exclude:**

| ✅ Include | ❌ Exclude |
|---|---|
| Bash commands Claude can't guess | Anything Claude can figure out by reading code |
| Code style that differs from defaults | Standard conventions Claude already knows |
| Testing instructions / preferred runners | Detailed API docs (link instead) |
| Repo etiquette (branch naming, PR conventions) | Info that changes frequently |
| Architectural decisions specific to the project | File-by-file descriptions |
| Env quirks (required env vars) | Self-evident advice ("write clean code") |
| Common gotchas / non-obvious behaviors | Long explanations or tutorials |

**Give Claude a map**: a project summary + high-level directory tree + main dependencies, architectural patterns (DDD/microservices), and non-standard organizational choices.

**Document custom tools** with usage examples and when to invoke them; mention `--help` so Claude knows to check it. For MCP servers, document constraints in CLAUDE.md (e.g. "Slack MCP: posts to #dev-notifications only; deploy/build failures only, not PR updates; rate-limited 10/hr"). Use `.mcp.json` for checked-in config; `--mcp-debug` to troubleshoot.

**Define a default workflow** — four questions before changing code: (1) is this a current-state question needing investigation first? (2) does it need a detailed plan? (3) what info is missing? (4) how will effectiveness be tested? Name workflows explicitly: explore-plan-code-commit for features, TDD for algorithmic work, visual iteration for UI.

**Security**: never put API keys, credentials, connection strings, or detailed vulnerability info in CLAUDE.md — it becomes part of the system prompt; treat it as potentially public.

**Compaction control**: add instructions like "When compacting, always preserve the full list of modified files and any test commands."

## 5. Rules (`.claude/rules/`)

Markdown files giving Claude specific constraints. **Path-scoped rules** load only when relevant via `paths:` frontmatter:

```yaml
---
paths:
  - "src/api/**"
  - "**/*.handler.ts"
---
All API handlers must validate input with Zod before processing.
```

A rule scoped to `src/api/**` stays out of context during a docs-only session. Reach for a path-scoped rule (over a nested CLAUDE.md) when the concern is cross-cutting — a file/constraint that appears in multiple but not all corners of the codebase (e.g. "migrations are append-only"). An **unscoped** rule is mechanically identical to CLAUDE.md content: always loaded, always costing tokens.

## 6. Skills — reusable knowledge & workflows

A skill is a **folder** (not just markdown): `SKILL.md` (required) + `scripts/` (executable) + `references/` (docs loaded as needed) + `assets/` (templates/fonts/icons). Skills are portable (Claude.ai, Claude Code, API) and composable. Bundled skills: `/code-review`, `/batch`, `/debug`. Skills **hot-reload** — edit a `SKILL.md` and the change is picked up without restarting.

**Three-level progressive disclosure:**
1. **YAML frontmatter** (always in system prompt) — just enough for Claude to know *when* to use it.
2. **SKILL.md body** (loads when relevant) — full instructions.
3. **Linked files** (navigated only as needed) — `references/api.md`, `stuck-jobs.md`, templates in `assets/`.

**The description field is a trigger specification, written for the model, not humans.** Only name + description influence triggering. MUST include BOTH *what it does* and *when to use it* (trigger phrases users actually say, file types if relevant); under 1024 chars; no XML angle brackets (frontmatter is injected into the system prompt — injection risk); structure = [verb + object] + ["Use when:" triggers] + [key capabilities] + [negative boundary]. Include **negative boundaries** ("Not for simple viewing… Do NOT use for X, use Y instead").
- *Good*: "Comprehensive PDF toolkit for extracting text/tables, creating, merging/splitting, handling forms. Use when filling a PDF form or processing/generating/analyzing PDFs at scale. Not for simple viewing or basic conversions."
- *Bad*: "Helps with PDFs and documents."
- **Debug triggering**: ask Claude "When would you use the [skill] skill?" — it quotes the description back; adjust what's missing. Under-triggering → add detail/keywords (esp. technical terms). Over-triggering → add negative triggers, be more specific.

**Naming (hard rules)**: `SKILL.md` exactly (case-sensitive); folder + `name` in kebab-case (no spaces/underscores/capitals); name matches folder. No `README.md` inside the skill folder (a repo-level README for humans is separate). Names with "claude"/"anthropic" prefix are reserved. Optional frontmatter: `license`, `compatibility` (1–500 chars), `allowed-tools` (e.g. `Bash(python:*) Bash(npm:*) WebFetch`), `metadata` (author/version/mcp-server/category/tags). **Note:** `allowed-tools` does *not* sandbox — it's an allow-list of what the model may reach for, not a security boundary; **`disallowed-tools` is the one that actually restricts** (see §16).

**Authoring craft:**
- **Don't state the obvious** — Claude already knows how to code. Knowledge skills should push Claude *out* of its default patterns (the frontend-design skill avoids Inter font + purple gradients).
- **The Gotchas section is the highest-signal content.** Build from real failure points, update over time: "table is append-only — take the highest version, not the latest `created_at`"; "field is `@request_id` in the gateway, `trace_id` in billing — same value"; "staging returns 200 even when the webhook didn't process."
- **Avoid railroading** — too-specific instructions hurt reusability; give info + flexibility.
- **MANDATORY-READ pattern** — when a reference file must be fully absorbed (API syntax, formatting rules), instruct "MANDATORY — READ ENTIRE FILE… NEVER set any range limits when reading this file" to stop Claude skimming. (From the docx skill, which reads its ~500–600-line reference files in full before generating.)
- **Dynamic injection** — a skill body can run `!command` to inject live output (e.g. current git branch, today's date) at invocation time.
- **Batching for large mechanical operations** — group related changes into batches of 3–10, test each batch before the next, make minimal precise edits (mark only what changes). Grep the source immediately before each script since line numbers shift after edits; verify at the end (old phrase should NOT appear, new should).
- **Setup via `config.json`** in the skill dir; if unset, the agent asks the user (use AskUserQuestion for structured choices).
- **Skills can have memory** — append-only log / JSON / SQLite (e.g. `standup-post` keeps `standups.log` to know what changed since yesterday). Use `${CLAUDE_PLUGIN_DATA}` for a stable data dir.
- **Give Claude code** — helper scripts/libraries let Claude spend turns on composition rather than reconstructing boilerplate.
- **On-demand hooks in skills** — active only for the session when invoked: `/careful` (PreToolUse blocks `rm -rf`, `DROP TABLE`, force-push, `kubectl delete`), `/freeze` (blocks edits outside a directory during debugging).
- **Deterministic validation**: for critical checks, bundle a script that performs them programmatically — "code is deterministic; language interpretation isn't."
- **"Menu" approach to size**: SKILL.md describes what's available and uses relative paths to separate files; Claude reads only the task-relevant one. Keep SKILL.md under ~5,000 words. If 20–50+ skills are enabled simultaneously, responses degrade — selective enablement / skill "packs."
- **Put critical instructions at top** with `## IMPORTANT` / `## CRITICAL` headers; for model "laziness," encouragement works better in the user prompt than in SKILL.md.

**When to build one**: the 5×/10× test — you've done the task ≥5 times and will do it ≥10 more. Iterate on a *single challenging task* until Claude succeeds, then extract the winning approach into a skill (leverages in-context learning, faster signal than broad testing). Single-purpose granularity: "SEO optimization for blog posts" is right; "Content marketing helper" too broad; "Add meta descriptions" too narrow. Include success criteria *in the skill* so Claude can self-check. **A skill is "intent written on the outside"** — without it, an agent re-derives your conventions (and guesses at the gaps) from zero every session; with it, knowledge compounds.

**Two skill types** (testing implications differ): **capability uplift** (does something the base model can't do consistently — may become obsolete as models improve; evals tell you when) vs **encoded preference** (sequences steps per your process — durable, only as valuable as its fidelity to your workflow).

**Nine internal categories** (best skills fit cleanly into one; straddling several confuses the agent): library/API reference, **product verification** (highest measurable impact on quality — worth a week to perfect; record video of output, assert state at each step), data fetching/analysis, business-process automation, code scaffolding/templates, code quality/review, CI/CD/deployment, runbooks, infrastructure operations.

**Five orchestration patterns**: sequential workflow (explicit ordering + rollback), multi-MCP coordination (phase separation + data passing), iterative refinement (quality criteria + know when to stop), context-aware tool selection (decision tree), domain-specific intelligence (embedded expertise, e.g. compliance-before-action). **Problem-first vs tool-first** framing (Home Depot analogy): problem-first → the skill orchestrates the right MCP calls; tool-first → the skill teaches optimal workflows for connected tools.

**Test at three areas**: triggering (fires on obvious + paraphrased; stays dormant on unrelated/out-of-scope), functional (valid outputs, API calls succeed, edge cases, error handling), performance comparison (with vs without skill — tool calls, tokens, retries).

**skill-creator** can write evals, run **benchmark mode** (pass rate, time, tokens), run evals via **parallel agents** in clean contexts (no cross-contamination), use **comparator agents** for blind A/B (skill vs no-skill), and tune descriptions to cut false positives/negatives. Results stay local — store, dashboard, or wire into CI.

**Distribution**: check into `./.claude/skills` for small teams; internal plugin marketplace at scale; org admins can deploy workspace-wide; API via `POST /v1/skills` (beta header `anthropic-beta: skills-2025-10-02`, requires Code Execution Tool beta). A **plugin** is how you *ship* a skill across repos (bundle skills + connectors). Agent Skills is an **open standard** (the same SKILL.md format works in Codex too, and MCP connectors are largely portable between the two). Public repos: `anthropics/skills`, Partner Skills Directory (Asana, Atlassian, Canva, Figma, Sentry, Zapier). Measure usage with a PreToolUse logging hook to find popular vs under-triggering skills.

## 7. Subagents & parallelism

A subagent is an **isolated Claude instance** with its own fresh context window — "the browser tabs of a session": chase a tangent without losing the main thread. It does the work and returns **only the result** (summary + metadata). Each starts fresh — no conversation history or invoked skills — and can have its own permissions (research = read-only, implementation = full edit), system prompt, tools, and model. They nest up to 5 levels deep; orchestration plan + intermediate results live in script variables, not context. **Subagents default to Sonnet**; you can override model, count, background mode, `maxTurns`, and `isolation: worktree` (each helper gets a fresh self-cleaning checkout) per agent.

**Startup loads**: the agent's own system prompt (not the full Claude Code one), preloaded `skills:`, and whatever the lead passes. **Inheritance differs by type**: general-purpose and **Plan** agents inherit the main context (CLAUDE.md + git status); **Explore** starts fresh (no CLAUDE.md/git) for fast read-only search — often on a cheaper model like Haiku.

**The mental test**: *"Will I need this tool output again, or just the conclusion?"* Only the conclusion → subagent.

> **An Explore/research subagent's summary is lossy.** When the details matter (you'll edit those files next), have the **main agent re-read the key files itself** rather than trusting the summary — the main agent's cross-attention over the actual source beats a compressed hand-off. Use subagents to *locate* and *triage*, not to be the sole source of truth for code you're about to change.

**When to use (signals):** gathering context needs reading dozens of files; sub-tasks have no dependencies (→ parallel); a fresh/unbiased perspective is needed; a second opinion before committing (catch overfitting to tests); pipeline stages with clean handoffs. **Quantitative threshold**: 10+ files to explore OR 3+ independent pieces of work.

**Subagent vs `/clear` for fresh eyes**: both reset bias, but a subagent keeps the main conversation intact while `/clear` loses history.

**Invocation, escalating** (start conversational, automate as patterns clarify):
1. **Conversational** — scope clearly, request parallelism explicitly, name the return format, ask for blind context. Example:
   ```
   Use subagents to explore this codebase in parallel:
   1. Find all API endpoints and summarize their purposes
   2. Identify the database schema and relationships
   3. Map out the authentication flow
   Return a summary of each, not the full file contents.
   ```
2. **CLAUDE.md policy** — defines *when* to reach for them ("When asked to review code, ALWAYS use a read-only subagent"). Custom subagents define *who*.
3. **Custom subagents** (`.claude/agents/`, project; `~/.claude/agents/`, user) — created via `/agents`. The **description field is the trigger spec** ("reviews code for security issues before commits" routes better than "security expert"). Tell it to say so explicitly if it finds nothing rather than inventing issues.
4. **Skills / hooks** — repeatable multi-step → skill (`/deep-review` spawns parallel security/performance/style reviewers); event-driven → hook.

`Ctrl+B` backgrounds a slow subagent (conversation continues, results surface automatically); `/tasks` shows background work.

**Five practical patterns**: research-before-implementing; parallel modifications (each follows a reference file's pattern); independent review (blind + critical); pipeline (output files as the handoff between design → implement → test stages); and the **maker/checker split** — one agent (or model) builds, a different one with different instructions verifies against the spec, because a model grading its own work is too lenient. A common artifact-based split that avoids merge conflicts: one **Implementer** (touches `src/`), one **Tester** (touches `tests/`), one **Documenter** (touches `docs/`) working the *same* task concurrently. Confirm the decomposition strategy with the user **before** spawning the fan-out.

**When NOT to use**: sequential dependent work (a relay through files is messier than one session); same-file parallel edits (conflict); small tasks (overhead > benefit); too many specialist agents (floods Claude, hurts auto-delegation — settle on a handful); work needing inter-agent coordination → **agent teams**.

**Agent teams vs subagents**: subagents report only to the main agent (lower cost). **Agent teams** are independent sessions with peer-to-peer messaging + a shared task list (higher cost — each teammate is a separate instance). Enable with `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` (experimental, off by default). Transition point: parallel subagents hitting context limits, or subagents needing to talk to each other. Best for research with competing hypotheses, parallel review, feature work where each teammate owns a piece.

**Parallelism economics (from 4.8's multi-agent benchmarks):** a fixed team of 5 agents (5M tokens total) beat a single agent (10M tokens) on a hard search benchmark while using **~20% of the latency** — but at **higher total token cost** (you trade tokens for wall-clock). Crucially, the speedup is **concentrated on the hard tail**: on easy problems coordination overhead cancels the gains (≈0 speedup); on the hardest problems the median speedup was ~3×. **So parallelize the genuinely hard, independent work — not everything.**

**Parallel session approaches**: `--worktree <name>` (isolated git checkouts, `.worktreeinclude` for cleanup); desktop app (visual, each in its own worktree); Claude Code on the web (isolated VMs); agent teams (coordinated). A fresh context also improves code review — Claude won't be biased toward code it just wrote. Use a **Writer/Reviewer** pattern across two sessions, or one Claude writes tests and another writes code to pass them. Before starting work in a new worktree/checkout, run the **baseline ritual** (`install && lint && test`) to confirm a clean starting point. Expect **10–20% of parallel sessions to be abandoned** to unexpected scenarios — that's a normal fan-out waste rate. Heavy users keep ~5 numbered terminal tabs + 5–10 web sessions; hand off local↔web with `&` / `--teleport`. The mechanical collision is solved by worktrees; **your review bandwidth is the real ceiling** on how many you can run.

## 8. Hooks — deterministic automation

Hooks fire on lifecycle events (tool execution, session boundaries, prompt submission, permission requests, compaction) and run a command, HTTP request, MCP tool, prompt, or subagent. The first three are deterministic; prompt/agent use model judgment. They bypass compaction and cost zero context unless they return output. Claude can write hooks for you ("write a hook that runs eslint after every file edit"). Configure in `.claude/settings.json`; browse with `/hooks`. A `PreToolUse` hook can also **modify a tool call's inputs** before it runs, not just allow/deny it.

- **PostToolUse format hook** (prevents CI failures from formatting drift):
  ```json
  "PostToolUse": [
    { "matcher": "Write|Edit",
      "hooks": [ { "type": "command", "command": "<your formatter> || true" } ] }
  ]
  ```
  > Caveat: a format-on-save hook **mutates files mid-session and can invalidate the prompt cache** for the rest of the conversation (changed bytes downstream of the cached prefix). If you see cache-hit rate collapse, this is a prime suspect — scope the matcher tightly or format on commit instead.
- **Stop hook that gates on tests** (blocks the turn from ending until tests pass; `stop_hook_active` guard prevents infinite loops):
  ```bash
  #!/bin/bash
  INPUT=$(cat)
  STOP_HOOK_ACTIVE=$(echo "$INPUT" | jq -r '.stop_hook_active // false')
  if [ "$STOP_HOOK_ACTIVE" = "true" ]; then exit 0; fi
  if ! npm test --silent > /dev/null 2>&1; then
    jq -n '{decision:"block", reason:"Tests are failing. Run `npm test`, fix them before finishing."}'
    exit 0
  fi
  exit 0
  ```
  > Make the gate **deterministic** (a real test run + exit code), not an LLM judging the transcript — 4.8 has been observed gaming an LLM-grader stop-hook by flooding its window with "PASSED" text (§0a).
- **`PreToolUse` exit code 2** to deny a tool call (real guardrail). **Put guardrails in hooks** ("never edit `.env`" as an instruction is a request; a PreToolUse hook is enforcement).
- **Context-survival hooks**: **`PreCompact`** backs up history before compaction; **`SessionStart`** re-injects key on-disk state (the current `plan.md`, a learnings file) so a fresh/compacted session reloads it automatically.
- **Auto-half-clone / context-watchdog Stop hook**: a `check-context.sh` that, when context exceeds ~85%, deterministically forks a fresh session carrying only the later (still-relevant) half — more predictable than waiting for autocompact.
- **Self-created notifications**: ask Claude to play a sound when done and it builds its own hook-based notification.

## 9. Session & context management — every turn is a branching point

After any completed turn you have five options; four exist purely to manage context:

| Situation | Reach for | Why |
|---|---|---|
| Same task, context still relevant | **Continue** | Everything in the window is load-bearing; don't pay to rebuild it |
| Claude went down a wrong path | **Rewind** (Esc Esc / `/rewind`) | Keep the useful file reads, drop the failed attempt, re-prompt with what you learned |
| Mid-task, bloated with stale exploration | **`/compact <hint>`** | Low effort; Claude decides what mattered; steer with instructions |
| Starting a genuinely new task | **`/clear`** | Zero rot; you control exactly what carries forward |
| Next step yields lots of output you'll only need the conclusion of | **Subagent** | Intermediate noise stays in the child; only the result returns |

- **New task = new session** (rule of thumb). Exception: related work where context is still load-bearing (writing docs for a feature you just built — a fresh session re-reads the files, slower + more expensive).
- **Rewind vs branch vs fork**: **Rewind** (Esc Esc) drops the dead end and replaces it. **`/branch`** (a.k.a. `/fork`) keeps *both* paths so you can compare approaches. **`/fork --fork-session`** (half-clone) starts a new session seeded with only the later, still-relevant half of the current one. Checkpoints persist across sessions; they track only Claude's changes (not a git replacement).
- **`/handoff` / "summarize from here"** creates a hand-off — a note from future-Claude to its past self about what was tried; write it to disk *before* you `/clear`.
- **`Ctrl+S` stashes the current prompt** (park a half-written instruction without losing it); **`/copy`** puts the last response on the clipboard as markdown.
- **Compact vs Clear**: Compact = lossy model summary, low effort, steerable (`/compact focus on the auth refactor, drop the test debugging`). Clear = you write the brief, more work but you control what carries forward. A good `/compress-session` keeps a "Discarded as noise" section so the same dead ends aren't re-summarized back in (anti-context-poisoning).
- **Bad autocompact** happens when the model can't predict where your work is going (a debugging summary drops the "other warning" you then ask to fix). Mitigate by compacting proactively with a direction hint while you still have room.
- **After two failed corrections on the same issue, `/clear`** and write a better prompt incorporating what you learned. A clean session with a better prompt almost always beats a long session of accumulated corrections.
- Name sessions with `/rename` and treat them like branches; `claude --continue` (most recent) / `claude --resume` (pick from list) / `/resume`. A `gh pr create` auto-links the session; return via `claude --from-pr <number>` or paste the PR URL into `/resume` search.
- **Put anything worth keeping on disk before compaction can erase it — "the agent forgets, the repo doesn't."** The compaction-survivable layers (Sant'Anna's four): `CLAUDE.md` (rules/structure/commands, re-read after compaction), `plan.md`/`SPEC.md` (task strategy, survives the planning→implementation handoff), `research.md` (system understanding), and **session logs** (the *why* behind decisions). The conversation window itself is ephemeral. After any compaction, "Read plan.md, that's our current plan" restores context instantly.
  - *Optional convention (Sant'Anna):* a committed `MEMORY.md` (≤200 lines, team-shared) and/or a gitignored personal memory file holding `[LEARN:category]` correction notes. This overlaps with the `#`-key-into-CLAUDE.md mechanism — **pick one**, don't run both. Promote a note to the durable file only after it's proven itself.

## 10. Workflow: explore → plan → code → commit

Letting Claude jump straight to coding produces code that solves the wrong problem. Separate research/planning from execution. This is "waterfall in fifteen minutes": a fast, disposable up-front spec/plan that the agent then executes — the discipline of waterfall at the speed of vibe-coding.

1. **Explore** (plan mode — reads files, no edits): "read /src/auth and understand how we handle sessions and login."
2. **Plan** (`Ctrl+G` opens the plan in your editor to edit before proceeding): "I want to add Google OAuth. What files change? What's the session flow? Create a plan." Iterate the plan 2–3 rounds; a good plan often one-shots execution. Spin up a second Claude to review the plan as a staff engineer.
3. **Implement** (switch out of plan mode): "implement the OAuth flow from your plan. write tests for the callback handler, run the suite, fix failures."
4. **Commit**: "commit with a descriptive message and open a PR." Treat commits as **save points** — commit liberally so any step is a clean checkpoint, and **never commit code you couldn't explain to someone else.**

Enter plan mode with `Shift+Tab` (twice) mid-session, or `claude --permission-mode plan`. **Skip planning** when you could describe the diff in one sentence (typo, log line, rename). Plan when uncertain about approach, when it touches multiple files, or when you're unfamiliar with the code.

**Research-language matters.** For existing code, use words like "deeply," "in great detail," "intricacies," "go through everything" — without them Claude skims at the signature level and moves on. Have it write findings to `research.md` as your review surface, and verify it understood the system before any planning (garbage in, garbage out — **the most expensive failure mode is code that works in isolation but breaks the surrounding system**, so make the agent "reference existing code constantly," not just the new file). Stash these explanations; they make excellent context to paste into future prompts.

**The annotation cycle (highest-leverage technique).** Use a real `plan.md` (not just built-in plan mode) as shared mutable state: open it in your editor, add inline notes directly where Claude is wrong ("not optional", "this should be a PATCH not a PUT", "the queue consumer already handles retries — remove this"), send back "I added notes to plan.md, address all of them and update the document. **Don't implement yet.**", and repeat 1–6 rounds until it's right. The "don't implement yet" guard is essential — without it Claude jumps to code the moment it thinks the plan is good enough. A real file persists and survives compaction in full fidelity (built-in plan mode is fine for the initial brainstorm; export to markdown for the annotation loop). Then extract a todo list: "Add a detailed todo list to the plan, all phases and tasks. Don't implement yet." — it becomes your progress tracker, and **the live todo list doubles as attention-management**: re-surfacing the remaining items each turn counters "lost-in-the-middle" drift on long runs.

**Item-level steering** (don't accept or reject a whole plan): cherry-pick the good parts, trim scope on an over-ambitious item, "protect" interfaces you don't want touched, and override specific choices — all as inline notes. If a direction is wrong, **revert-and-re-scope** rather than patching forward.

**Reference-implementation trick:** paste a known-good external/OSS snippet into the plan as the reference pattern ("build it like this") — far more reliable than describing the pattern in prose.

**The implementation command** (reusable, once the plan is approved):
> "Implement it all. When you finish a task or phase, mark it complete in plan.md. Do not stop until all tasks and phases are done. Do not add unnecessary comments or jsdocs. Do not use any/unknown types. Continuously run typecheck to make sure you're not introducing new issues."

Implementation should be **boring** — all creative decisions happened during annotation. If implementation feels creative, your plan wasn't done.

**Disposable-plan convention** (for teams): one plan file per PR under `.ai/plans/`, named by ticket (`JIRA-1234-1.md`, `-2.md`…), written so *another agent could execute it autonomously*, then deleted/archived once merged. An **Expert Task Decomposer** prompt is a good front door: "Propose task titles only, then STOP for my approval; on approval, expand each into Task / Problem / Dependencies / Plan / Success-Criteria; bias toward more, smaller, atomic tasks." For very large specs, have the agent build an **extended TOC / spec-summary** (each section → 1–2 sentences + a reference tag) to keep a "mental map" in context while full sections load on demand.

**Let Claude interview you for larger features** — start minimal (a "brain dump" / context-packing pass where you dump everything relevant, including approaches to *avoid*), have it interview you with AskUserQuestion (one question at a time, prefer multiple choice), then write a self-contained `SPEC.md` (names files/interfaces, states what's out of scope, ends with an end-to-end verification step). Start a **fresh session** to execute it.

**The architect/implementer dual-session loop** (high-reliability build pattern):
- Architect session brainstorms the design (describe it back in 200–300 word chunks, confirm each — capping output forces you to actually read it) → writes to `docs/plans/`.
- **Zero-context-engineer planning prompt**: "Assume the engineer has zero context for our codebase and questionable taste. Document everything: which files to touch per task, code, testing, docs to check, how to test. Bite-sized tasks. DRY. YAGNI. TDD. frequent commits. Assume a skilled dev who knows almost nothing about our toolset/domain and doesn't know good test design."
- Implementer (separate session): "Execute the first 3–4 tasks. If you have questions, stop and ask. DO NOT DEVIATE FROM THE PLAN."
- Architect: "The implementer says it's done tasks 1–3. Check the work carefully." (Double-Esc the architect to a prior checkpoint before each review so it re-reviews without bias.)
- **`/clear` (not `/compact`) the implementer between chunks**; restart at "task 4." The **plan doc is the persistent handoff artifact** — the implementer updates it with current state after each signed-off chunk.

**Input-box shortcuts worth knowing** (readline): `Ctrl+A`/`Ctrl+E` (line start/end), `Ctrl+W` (delete word), `Ctrl+U`/`Ctrl+K` (kill to start/end), `Ctrl+G` (open the prompt/plan in `$EDITOR`).

**Onboarding to a codebase**: ask the questions you'd ask a senior engineer — "how does logging work?", "how do I make a new API endpoint?", "what does `async move {...}` do on line 134?", "what edge cases does `CustomerOnboardingFlowImpl` handle?". Ask for a glossary of project-specific terms. Tests are a great onboarding tool (Claude reads them to learn a feature).

**Everyday recipes**: bug-fix (give the reproduce command + stack trace; note intermittent vs consistent; **paste the raw error/log straight in** — `cat error.log | claude` — rather than describing it). Refactor (small testable increments, maintain backward compat, verify with tests). Tests (Claude matches existing style/framework; ask it to identify edge cases — error conditions, boundary values, unexpected inputs). PRs ("create a pr"; enhance the description with context; review before submitting).

## 11. Prompting

- **Delegate, don't pair-program.** Treat Claude like a capable engineer you delegate to. **Front-load the full task in the first turn**: intent, constraints, acceptance criteria, file locations. Every user turn adds reasoning overhead, especially late in long sessions — batch your questions. Ambiguous prompts dripped across many turns hurt both token efficiency and quality.
- **Be specific**: scope the task (which file, what scenario, testing prefs — "write a test for foo.py covering the logged-out edge case, avoid mocks"), point to sources ("look through ExecutionFactory's git history and summarize how its api came to be"), reference existing patterns ("look at how widgets are implemented on the home page; HotDogWidget.php is a good example; follow the pattern"), describe the symptom ("login fails after session timeout. check src/auth/, especially token refresh. write a failing test that reproduces it, then fix it"). Vague prompts are fine when exploring ("what would you improve in this file?"). **Vague prompts are the #1 reason agent runs fail** — "you are a helpful coding assistant" does nothing; "you are a test engineer who writes tests for React components, follows these examples, and never modifies source" works.
- **Provide rich context**: `@file` (includes full content; also pulls in CLAUDE.md from that dir + parents), `@dir` (listing, not contents), `@server:resource` (MCP). Paste images (drag-drop, or `ctrl+v` — NOT cmd+v); ≥50% of some practitioners' prompts include a screenshot — the model matches strings and jumps to the exact location. Pipe data (`cat error.log | claude`, `git log --oneline -20 | claude -p "summarize"`). For "explain this whole repo" tasks, a packed dump via **gitingest / repo2txt** is a fast way to hand over a codebase. Tell Claude to fetch its own context (Bash, MCP, reading files).
- **Provide goals, not instructions** — if the agent can evaluate success, let it run.
- **Positive examples of the voice you want beat negative "don't do this" instructions.** State length/style explicitly if you need a specific one (the model calibrates length to complexity by default, and 4.8 trends long — ask for concise).
- **Want more tool use / more subagents?** Spell it out (the model calls tools less and spawns fewer subagents by default at higher quality): "Do not spawn a subagent for work you can complete directly in a single response. Spawn multiple subagents in the same turn when fanning out across items or reading multiple files."
- **Trigger words for hard problems**: "take your time," "comprehensive," "read all code that could be related," "create possible hypotheses."
- **Two four-word power-prompts** that encode large discipline: **"First run the tests"** (forces the agent to learn the test command → near-certain to re-run later; gauges project size; induces a testing mindset) and **"Use red/green TDD"** (write tests first, confirm they fail, **commit the tests**, then implement to green — guards against non-working and dead code; committing the tests first stops the agent quietly editing them to pass).
- **By-example direction ("Draw the Owl")**: make a change illustrating intent and ask the agent to finish, or do the thing fully once and tell it to read that commit and repeat the pattern.
- **Throw-away the first draft.** Treat the first attempt at anything genuinely novel as a prototype to learn from, then discard it: "knowing everything you know now, scrap this and implement the clean solution."
- **Break a large task into smaller ones** (A → A1, A2, then B) rather than one monolithic prompt — both for the context window and the "curse of instructions" (§4). One focused task with only the relevant context beats everything-at-once.
- **Adversarial prompting** (single session): "grill me on these changes and don't make a PR until I pass your test"; "prove to me this works" (diff main vs branch); after a mediocre fix, "knowing everything you know now, scrap this and implement the elegant solution."

## 12. Model, effort & prompt caching

(See §0 for this repo's chosen config; §0a for 4.8's behavioral profile.)

**Prompt caching is the dominant cost/latency lever** and is a **prefix match** — any change anywhere in the prefix invalidates everything after it. The harness orders content static→dynamic to maximize cross-session hits: static system prompt + tools (globally cached) → CLAUDE.md (per project) → session context (per session) → conversation messages. Practical consequences for you:
- **Don't switch models mid-session** — caches are model-unique; switching Opus→Haiku 100k tokens in is *more* expensive than letting Opus answer (the Haiku cache must be rebuilt). If you must switch, use a subagent with a hand-off message.
- **Don't add/remove tools mid-session** — tools are part of the cached prefix; connecting/disconnecting an MCP server mid-session invalidates the whole conversation cache. (This is why Plan Mode keeps all tools loaded and models the state via `EnterPlanMode`/`ExitPlanMode` tools; and why MCP **tool search** uses `defer_loading` stubs instead of removing tools.)
- **Watch the cache TTL.** The default cache window tightened (≈60 min → ≈5 min in early 2026); enable the longer window with `ENABLE_PROMPT_CACHING_1H` for slow, bursty sessions, and diagnose unexpected misses with the `cache_miss_reason` beta field. A **format-on-save hook can silently invalidate the cache** (§8).
- Compaction uses **cache-safe forking** (same system prompt/tools/history as parent), reserving a "compaction buffer."

**Tool/model routing (70/20/10 rule of thumb):** keep the main session on the strong model and route by difficulty — roughly **~70% of bulk/boilerplate work to a cheap model (Haiku), ~20% to a mid model (Sonnet), ~10% to Opus** for the genuinely hard reasoning. Use a cheaper/faster model for first drafts and repetitive expansions; reserve the strong model for design and hard reasoning. (`/model opus[1m]` / `sonnet[1m]` selects the 1M-context variants when you actually need them.)

**Lazy-load MCP tools** with `ENABLE_TOOL_SEARCH` (the harness defers tool definitions and searches them on demand; auto-engages once MCP tools exceed ~10% of context) — keeps a big connector roster from eating your window.

**Economics**: for heavy use, subscription plans run roughly 5–10× cheaper than API pricing for the same token volume — and a high cache-hit rate is what makes generous subscription rate limits possible, so the cache-preserving habits above also directly protect your effective throughput.

## 13. Environment design — make the codebase agent-friendly

The agent is only as good as the tools and signals you give it. **Anything observable or interactable is a tool** (a shell script, a log file, an MCP server).

> **Amdahl's-law framing (why this section matters more now):** making code-*writing* fast just moves the bottleneck to everything around it — figuring out what to build, manual click-through, human review, and safe deploy. Agentic coding *exposes* underinvestment in your build/CI/review/deploy loop. The fix is to invest in tooling and feedback speed, not to "use the AI more." A fast CI that feeds its own output back to the agent is a force-multiplier; a slow one is the new ceiling.

**Five tool rules**: tools must be **fast** (crashes tolerable, **hangs are fatal**); **user-friendly** (clearly signal misuse/errors so the agent makes forward progress); **protected against an "LLM chaos monkey"** (no undefined behavior); **observable/debuggable**.

**The Makefile + file-logging pattern** (single highest-value environment tip):
- `make dev` starts services via a process manager. **Guard against double-spawn with a pidfile** — otherwise the agent, unsure whether the server is running, re-spawns it and binds two services to the same port.
- **Always log to a file, not just the terminal**, so the agent can `make tail-log` to diagnose. Side effects matter: log sign-in emails to stdout/file in debug mode so the agent can complete a full browser sign-in unaided (tell it via CLAUDE.md that emails are logged).
- **Speed**: emergent tools the agent writes itself need fast compile/execute (3ms vs 5s-compile-then-1min-boot changes the whole flow). If your stack is slow, vibe-code a hot-reload daemon that watches a dir, imports/executes modules placed there, and writes output to a log the agent can `cat`. Balance log verbosity (informative yet concise — token cost); give the AI knobs to control it. Ideal: observability as a byproduct of first-shot code generation beats a write→fail→add-debug loop.

**Feed machine output straight back in.** Pipe lint errors, failing CI logs, type-checker output, and review-bot comments directly back to the agent (`gh run view --log-failed | claude -p "fix these"`) instead of summarizing them — the raw output is the highest-fidelity signal.

**Write the dumbest code that works**: descriptive long function names over classes; avoid inheritance/clever hacks; **use plain SQL** (agents write excellent SQL and can match it against SQL logs — beats min-maxing an ORM); **keep permission/security checks local and visible** (hiding them in another file/config nearly guarantees the agent forgets them on new routes); keep functions at the **right level of abstraction** (one job each) so a diff is reviewable.

**Stability**: be **more conservative about dependency upgrades** with agents — they make upgrades "cheap" but invalidate earlier breadcrumb decisions/comments. Prefer code generation over adding dependencies; favor stable, low-churn ecosystems. Avoid `$` in filenames (shell interpolation mangles `$param.tsx` → `.tsx`).

**Refactor at the right moment** — not too early, not too late ("too big" = total the agent must consider). Dedicated refactor phases (great when low-focus) using **jscpd** (duplication), **knip** (dead code), eslint plugins, consolidating routes, splitting oversized files, finding/rewriting slow tests, modern patterns. Consider **ast-grep** as a codebase linter wired into a commit-blocking git hook. A "simplify this overcomplicated code" pass (and removing prose-repetition / redundant comments) is a cheap recurring win.

**Agentic manual testing mechanisms** (passing tests ≠ works): Python `python -c "..."` for edge cases; write throwaway code in `/tmp` to avoid accidental commits; JSON APIs via "run a dev server and explore that API using curl" ("explore" covers lots of ground); web UIs via **Playwright** (accessibility-tree based — good for assertions on structure) vs the **native `/chrome` integration** (screenshots + pixel coordinates — good for visual/layout checks); agent-built CLI wrappers; or `chrome-devtools-mcp` to close the loop; "look at screenshots" cues the agent to use its own vision. When manual testing finds a bug, fix it with red/green TDD so it lands in the permanent suite. Use CLI tools (`gh`, `aws`, `gcloud`, `sentry-cli`) — the most context-efficient way to hit external services; teach Claude unknown CLIs via `--help`.

**Exploratory prototyping for tech choices**: "Is Redis right for this activity feed?" → have the agent build a simulation + load test; cheap enough to run several competing experiments at once.

**Blast radius**: before any change, estimate how long it'll take and how many files it'll touch. Throw many small bombs, or one "Fat Man" plus a few small ones — but *multiple* large bombs at once make isolated commits impossible and resets harder. Use it as a live signal: if work runs longer than its anticipated blast radius, hit `Esc`, ask "what's the status," then help/abort/continue (file changes are atomic — don't fear stopping mid-way). "Give me a few options before making changes" gauges impact when you're unsure. For parallel agents in one folder, instruct each to commit *only the files it edited* so history stays clean and agents don't revert each other.

## 14. Slash commands

Codify every inner-loop workflow you run many times a day. Markdown files in `.claude/commands/`, checked into git, support `$ARGUMENTS` or numbered `$1`/`$2`, and **inline `!`-prefixed bash** to pre-compute context (`!git status`) so the model doesn't spend a round-trip fetching it. Ask Claude to write them ("create a custom slash command called /performance-optimization that…"). Slash commands are **user-invoked** (you decide when); skills are **Claude-invoked** (it decides from the description) — same content, different trigger.
- **`/commit-push-pr`** uses inline bash to pre-compute `git status`, avoiding model round-trips.
- Useful set seen in practice: `/commit` (only commit your own files when multiple agents share a folder), `/automerge` (one PR at a time, react to bot comments, CI green, squash), `/massageprs` (automerge without squash for parallelism), `/review` (built-in), `/gha` (drive GitHub Actions), `/techdebt`, `/context-dump`.
- **Disable Claude's commit/PR attribution** via settings (commit trailer / PR body / session-URL) if you don't want the "🤖 generated with Claude Code" footer.
- A `.claude/commands/` file is the legacy flat single-file form; a skill directory is the richer successor (templates, examples, scripts).

## 15. Scaling & automation

- **Non-interactive mode**: `claude -p "prompt"` for CI, pre-commit hooks, scripts. Output formats: text, `--output-format json`, `--output-format stream-json --verbose` (real-time). `--allowedTools "Edit,Bash(git commit *)"` scopes permissions for unattended runs. (`DISABLE_AUTOUPDATER=1` for reproducible CI.)
- **Fan out across files**: have Claude list all files needing a change, loop `claude -p` over them, test on 2–3 first, then run at scale. Pipe into pipelines: `claude -p "<prompt>" --output-format json | your_command`.
- **Run on a schedule**: **Routines** (Anthropic infra, runs when your computer is off, triggers on schedule/API/GitHub events — claude.ai/code/routines); **Desktop scheduled tasks** (local file access); **GitHub Actions** (repo events/cron); **`/loop`** (polling in the current CLI session). Scheduled prompts must be explicit about success criteria — they can't ask clarifying questions.
- **Message queuing**: queue related feature tasks; the agent works them off. Queue "continue" messages to let a big refactor finish while you're away.
- **tmux as a control layer**: not just for background CLIs (dev servers, long tests — "run via tmux"), but as an orchestration surface — `new-session` / `send-keys` / `capture-pane` lets the agent drive `git bisect`, TTY-interactive tools, and even a *second* containerized worker-Claude (run that worker with `--dangerously-skip-permissions` inside the container only).
- **Phone / remote control**: `claude remote-control` exposes a session to the mobile app; `--spawn=worktree --capacity=N` lets you start fresh worktree-isolated sessions from your phone.
- **Polling external jobs cheaply**: manual exponential backoff (sleep 1m → 2m → 4m, re-check) is more token-efficient than streaming `gh run watch`; prefer **pull** (`gh run view`) over a push-watch that floods context. `gh api graphql` for queries the CLI doesn't expose directly.
- **`/powerup`** for interactive lessons; Claude has built-in access to its own latest docs regardless of version.

**Loop engineering (advanced — high leverage, high risk).** The frontier of agentic work is *designing the loop that prompts the agent* rather than prompting it yourself ("I don't prompt Claude anymore; I write loops that prompt Claude"). A loop is a recursive goal: define the purpose, let the system iterate until it's verifiably complete. Five building blocks plus memory, all now shipping in the tooling:
1. **Automations / scheduling** = the heartbeat (a `/loop` or Routine that does discovery + triage on a cadence and drops findings in an inbox; have it call a **skill**, not a pasted wall of instructions).
2. **Worktrees** so parallel runs don't collide.
3. **Skills** = durable project intent so the loop doesn't re-derive your conventions every cycle.
4. **Plugins / connectors (MCP)** so the loop can act in your real tools (open the PR, update the ticket, ping the channel).
5. **Maker/checker subagents** so a *different* agent verifies — the only reason you can walk away while it runs unattended.
6. **On-disk memory** (a markdown file or board) as the spine that remembers what's done and what's next across runs.

`/loop` = cadence; `/goal` = run until a *verifiable* condition holds (a separate small model checks "done" after each turn — maker/checker on the stop condition itself).

> **The three risks that get *sharper* as the loop gets better**, not easier: **(1) verification is still on you** — an unattended loop is an unattended mistake-maker; "done" is a claim, not a proof. **(2) Comprehension debt** — the faster it ships code you didn't write, the bigger the gap between what exists and what you understand; only reading the output closes it. **(3) Cognitive surrender** — it's tempting to stop having an opinion and accept whatever comes back. The same loop, in two hands, produces opposite results: a force-multiplier for someone who understands the work, a hole-digger for someone avoiding it. **Build the loop; stay the engineer.**

## 16. Security (non-negotiable)

**Permission model (know what actually restricts):**
- The permission stack has six layers (managed/admin policy, project settings, user settings, in-session grants, the permission mode, and per-tool allow/deny). **In-session grants and the mode (`Shift+Tab`) are authoritative for the running session.** `allowed-tools` is *not* a sandbox — it's an allow-list of what the model may reach for; **`disallowed-tools` / a deny-list is what actually blocks.**
- The five permission modes range from prompt-everything to bypass-all. **Plan → Bypass is a convenience handoff, not an enforcement boundary** — don't treat "it was in Plan mode" as a guarantee it can't act.

**Supply chain / operating safety:**
- **Vet every skill, MCP server, plugin, and project config** before use — hundreds of malicious skills have been found in the supply chain; malicious repo configs can abuse hooks + MCP for RCE when you clone untrusted repos. Inspect `.claude/` before running Claude Code in a cloned repo.
- MCP servers can read/write your whole codebase — scope their permissions.
- Pin dependencies in any AI-tooling environment; check for known-compromised versions (e.g. the LiteLLM `1.82.7–1.82.8` compromise — pin `litellm<=1.82.6`; check for a persistence indicator before trusting a fresh install).
- Prefer `/permissions` allowlists (`bun run build:*`, `bun run test:*`) and **auto mode** over `--dangerously-skip-permissions`; reserve the latter for long-running sandboxed tasks. Use `/sandbox` (OS-level isolation) or Docker to limit blast radius; run YOLO/no-approval mode only when malicious context can't sneak in. Async cloud agents for riskier tasks (worst case = source leak).
- Frontmatter is injected into the system prompt — no XML angle brackets in skill descriptions (injection vector).
- **The model is over-eager by default** — 4.8 specifically has been observed working around restrictions to satisfy the "overriding goal" (bypassing a network proxy, retrying after "do not retry," deleting files when only debatably necessary — §0a). Your `deny`-list + sandbox + a three-tier boundary block (§4: ✅ always / ⚠️ ask-first / 🚫 never) are the real safety net, not prose. Put auth, payments, and data-mutation operations in the ⚠️ "ask first" tier and **always review them yourself.**

**Prompt injection (especially for agents that read untrusted external content — scraped pages, emails, tool results, documents):**
- A prompt injection is a malicious instruction hidden in content the agent processes; a successful one makes the agent follow it as if it came from you. It scales — one payload in a public page or shared doc can hit every agent that reads it.
- 4.8 is **somewhat less robust to prompt injection than 4.7 in some agentic contexts** without product-level safeguards (its raw browser-use and computer-use robustness varies by surface). **Don't combine, unguarded, an agent that can read untrusted content with an agent that can take powerful actions** (exfiltrate data / mutate state). Isolate fetch-and-summarize from act-on-results; treat anything fetched from the web/an inbox as hostile by default. (Directly relevant to OSINT/scraping pipelines.)

**Write-time code security** (bake it in while Claude writes, not as an afterthought — AI-generated code introduces vulnerabilities a non-trivial fraction of the time without guidance, and AI-speed development means more flaws in the same window):
- **Add a security path-scoped rule** (`.claude/rules/`): no hardcoded secrets; parameterized queries (never string-concatenate SQL); validate/sanitize all inputs; **authorization checks on every endpoint, not just authentication**; no `eval()`/`exec()` with user input; security headers; safe deserialization (no `pickle`/unsafe YAML on untrusted input — a classic vibe-coding RCE).
- **Keep auth/permission checks local and visible** (also §13) — hiding them in another file/config nearly guarantees the agent forgets them on new routes.
- **Run a deep security review** at integration points and before deploy (scan for secrets, injection vectors, missing authz, IDOR): the bundled `/security-review`, or a `security-reviewer` subagent (§7), or the security-review GitHub Action on each PR (use a cheaper model for cost on automated runs).
- **Self-reflection prompting** reduces insecure output — ask the model to review each component it generates for security issues; prefer language/framework-specific guidance over generic.
- A dedicated security skill (e.g. the third-party **VibeSec** — community, not official Anthropic) can apply secure patterns automatically across access-control (IDOR, privilege escalation, mass assignment), client-side (XSS, CSRF, secret exposure, open redirect), server-side (SSRF, SQLi, path traversal, insecure upload), and auth (session management, password hashing, token revocation). Vet it like any third-party skill before use.

## 16a. Frontend — avoid generic "AI slop"

(Most relevant for any UI/visual work; lives here as a focused reference. Best encoded as a `frontend-design` skill or a path-scoped rule.)

Without guidance Claude converges on generic output: Inter/Roboto/Arial/system fonts, purple gradients on white, predictable layouts, minimal motion. To counter it:
- **Commit to a BOLD aesthetic direction before coding** — pick an extreme (brutally minimal, maximalist, retro-futuristic, brutalist/raw, editorial, art-deco/geometric, industrial/utilitarian…). Intentionality, not intensity — bold maximalism and refined minimalism both work.
- **Typography**: never Inter/Roboto/Arial/system fonts; pair a distinctive display font with a refined body font, use extreme weight contrast; don't even converge on the same "distinctive" choice (e.g. Space Grotesk) across generations.
- **Color**: cohesive CSS-variable system; dominant colors with sharp accents beat timid evenly-distributed palettes; never default purple-on-white.
- **Motion**: one well-orchestrated page load with staggered reveals beats scattered micro-interactions; CSS-only for HTML, Motion library for React.
- **Composition**: asymmetry, overlap, diagonal flow, grid-breaking, generous negative space *or* controlled density.
- **Backgrounds**: atmosphere/depth — gradient meshes, noise textures, layered transparencies, dramatic shadows, grain overlays — not solid colors.
- **Match implementation complexity to the vision** (maximalist needs elaborate code; minimalist needs restraint in spacing/typography). **Iterate visually 2–3 rounds** — the third pass is usually dramatically better. Verify visually: "[screenshot] implement this design. Take a screenshot, compare to the original, list differences, fix them."
- The highest-value design skills give Claude precise info it *doesn't* inherently have — exact hex codes, font names, size thresholds (e.g. "Poppins for headings 24pt+, Lora for body").

## 17. Where practitioners disagree (calibrate, don't cargo-cult)

The structured Anthropic playbook above is the default. Several experienced practitioners run leaner setups and dispute specific patterns. Both can be right depending on your stack and how you test.

- **Worktrees vs one folder.** Anthropic docs + many practitioners favor worktrees for isolation. Boris Cherny uses separate **git checkouts**, not worktrees. Some run **3–8 agents in one folder** with a single dev server, reverting from worktrees because clicking through one running app to test multiple changes is faster (multiple dev servers / branch-per-change is annoying; OAuth callback-domain limits). **Rule of thumb: if you test through one running app, one folder may beat worktrees; if changes are independent and headless, worktrees win.**
- **Subagents vs separate windows.** Subagents isolate context but reduce your visibility/control over what's sent back. Some prefer doing research in a **separate terminal pane** and pasting results, for full control of engineered context. A generic persona subagent ("You are an AI engineer specializing in...") adds no value and can be **context poison** — what helps is documentation, examples, and do/don't lists, not role strings. (Consistent with §6's "don't state the obvious.")
- **MCPs vs CLIs.** Most MCPs arguably should be CLIs: refer to a CLI by name, the agent hits its `--help`, and now has full usage in context at **zero permanent context tax** — vs an MCP's constant cost (the GitHub MCP can consume ~23k tokens; `gh` does the same job for free). Reserve MCP for things genuinely hard to use otherwise (browser automation) — and even then a curl-able API endpoint is often more token-efficient. A related advanced pattern: expose your tools as a small **code API** the agent calls in a code-execution tool, rather than as many individual tool definitions, to keep the tool schema out of the prompt.
- **Spec-driven vs conversational.** Heavy upfront specs suit some work and weaker models. With stronger models, some prefer: discuss → flesh out together → for tricky parts write a spec, send it to an *even stronger* model for review, paste useful bits back. For UI work, deliberately under-spec, watch it build live, and iterate/morph rather than reset. Religiously starting a fresh context with the plan is a weaker-model habit; strong models handle large contexts (re-fetching files costs minutes). **Single-long-session stance (Boris Tane):** some run one long session for a whole feature and report *no* degradation past 50% context — understanding accrues and `plan.md` survives compaction — which directly cuts against §2's context-rot caution. Reconcile by task: exploratory/iterative work tolerates long sessions; precision-critical work benefits from frequent fresh contexts.
- **Cross-model review ("model musical chairs").** Build with Claude, then have a *different* model (GPT/Gemini/Codex) review via `/review` or a paste — different blind spots catch different bugs (LLM-as-a-judge for the subjective stuff: style, readability, architecture). Cheap insurance on anything important.
- **Model-specific prompting.** Claude responds to emphatic ALL-CAPS / "IMPORTANT / YOU MUST"; some other models react badly to caps/threats and prefer plain human phrasing. Agent instruction files don't transfer cleanly across vendors.
- **Overstructure skepticism.** A recurring minority view: don't over-invest in RAG, elaborate subagent rosters, or "Agents 2.0" charade — "just talk to it, play with it, develop intuition." Treat the heavy machinery as tools to reach for when a trigger appears (§3), not scaffolding to build up front.

## 18. Failure & anti-patterns

- **The kitchen-sink session** — unrelated tasks pile up; context full of noise. → `/clear` between tasks.
- **Correcting over and over** — context polluted with failed approaches. → after two failed corrections, `/clear` + a better prompt.
- **Over-specified CLAUDE.md** — important rules lost in noise ("curse of instructions"). → ruthlessly prune; convert "always do X" to hooks.
- **Trust-then-verify gap** — plausible code that misses edge cases. → always provide verification; if you can't verify it, don't ship it. (Willison's rule: "I won't commit code I couldn't explain to someone else.")
- **Infinite exploration** — unscoped "investigate" reads hundreds of files. → scope narrowly or use subagents.
- **Inflicting unreviewed code on collaborators** — never file a PR with agent code you haven't reviewed yourself ("they could've prompted an agent themselves"). A good agentic PR: you're confident it works + small enough to review efficiently (several small PRs > one big; agents do the git splitting) + includes the higher-level goal/links + you've read the agent-written PR description. Include evidence of your own work (manual-test notes, screenshots, video). A final review on the **GitHub web UI** (a different surface/context from where you built) surfaces mistakes you'd miss in-terminal.
- **Agent credulity** — agents take pasted instructions (including external code-review comments) at face value and will make wrong changes. Mitigate with **hire/no-hire role-play**: "A reviewer analyzed this PR, reading the codebase cold. Evaluate the analysis AND the reviewer: 1) should we hire this reviewer 2) which flagged issues should be fixed 3) are their fixes correct? Anything to fix → todo list; anything to skip → tell me." Claude will sometimes "No hire" a reviewer that didn't understand project intent.
- **A reviewer prompted to find gaps will report some even when the work is sound** — tell it to flag only gaps affecting correctness or stated requirements; chasing every finding leads to over-engineering.
- **The "lethal trifecta" of agentic coding** — *speed* (it works faster than you can review), *non-determinism* (same prompt, different output), *cost* (which tempts corner-cutting on verification). Your spec + review process has to account for all three; don't let speed outrun your ability to verify.

**Prioritize review by feedback-loop length (impact-radius model).** Categorize where an agent's mistakes land and review accordingly:
- **Time-to-commit (caught fast):** obvious bugs, broken builds — cheap, the loop catches them.
- **Iteration / team-flow (caught medium):** broad-not-incremental changes, brute-forcing instead of root-causing, workflow complications.
- **Codebase-lifetime / maintainability (caught slow, most insidious):** redundant/brittle tests, lack of reuse / duplicated components, needlessly complex code, unnecessary parameters. **These rarely fail a test, so they need *human* eyes** — spend your review budget here, not on what CI already catches. (A weekly "where did the agent go wrong" log turns these into CLAUDE.md/skill updates.)

[ ] Apply the §1a behavioral guardrails every task: think-before-coding, simplicity-first, surgical diffs, goal-driven loops

[ ] Drop relevant skills into .claude/skills/ (verify each SKILL.md before use)
[ ] Path-scope cross-cutting constraints in .claude/rules/ (incl. a security rule: parameterized SQL, authz per endpoint, no eval/unsafe deserialization)

[ ] Use a fresh subagent (or second session) for review before committing; verify completion claims yourself; cross-model review for anything important
[ ] Press # to append a rule to CLAUDE.md after every Claude mistake (or keep one MEMORY.md — not both)
